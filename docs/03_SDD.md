# System Design Document (SDD)
**Проект:** NoCrumbs Cookie Notice (Минималистичный плагин уведомлений о cookie)
**Версия:** MVP

## 1. Архитектурный подход и Стек технологий
Решение проектируется с упором на максимальную минималистичность (Zero Bloat), безопасность и совместимость со всеми видами кэширования в WordPress (включая Edge/Page Cash).

*   **Backend (PHP 7.4+):** Использование нативных API WordPress (Actions/Filters, i18n, Options API). Реализация через ООП-паттерн с изоляцией логики в классах (без засорения глобального пространства имен).
*   **Frontend (HTML/Vanilla JS/CSS3):**
    *   JS: Стандарт ECMAScript 6+. Отсутствие jQuery. Использование нативного `document.cookie`.
    *   CSS: Нативные свойства, CSS-переменные не обязательны, Flexbox для выравнивания внутри карточки. Использование префикса `nc-` для строгой изоляции стилей (например, `.nc-cookie-banner`).

## 2. Структура файлов (File Structure)
Проект будет структурироваться по стандарту (согласно гайдлайнам WP Plugin Development, с единой точкой входа):

```text
nocrumbs-cookie-notice/
├── nocrumbs-cookie-notice.php     // Точка входа: комментарий-заголовок, инициализация классов.
├── includes/
│   └── class-nocrumbs-cookie-notice.php  // Главный сервис-класс плагина: регистрация хуков.
├── assets/
│   ├── css/
│   │   └── nocrumbs-cookie-notice.css  // Минифицированный CSS-файл плашки.
│   └── js/
│       └── nocrumbs-cookie-notice.js   // Логика клиентской стороны по проверке и записи cookie.
└── languages/
    └── nocrumbs-cookie-notice.pot      // Файл базовых переводов (i18n).
```

## 3. Общая логика (System Flow)

### Шаг 1: Инициализация (Backend)
1.  Событие `plugins_loaded`: подключается локализация `load_plugin_textdomain`.
2.  Событие `init`: загружается текстовый домен для переводов.
2.  Событие `wp_enqueue_scripts`: регистрируются и подключаются `nocrumbs-cookie-notice.css` и `nocrumbs-cookie-notice.js`.
    *   *Условие:* Подключение происходит **только если** `! is_admin()` (защита от исполнения в панели управления).
3.  Событие `wp_footer`: выполняется рендер HTML-каркаса плашки.

### Шаг 2: Формирование HTML-разметки (PHP -> Frontend)
1.  Сервер запрашивает ссылку на политику конфиденциальности с помощью ядра: `get_permalink( get_option('wp_page_for_privacy_policy') )`.
2.  Строится HTML-строка:
    *   Изолированный контейнер `<div id="nc-cookie-banner" class="nc-hidden" aria-hidden="true">`
    *   Текст: `esc_html__('Мы используем файлы с данными (cookie) для работы сайта.', 'nocrumbs-cookie-notice')`
    *   Ссылка "Политика конфиденциальности" (или пусто, если страница не настроена в WP).
    *   Кнопка `<button id="nc-cookie-accept">ОК</button>`.
3.  Разметка выводится строго перед закрывающим тегом `</body>`. Изначально плашке через CSS задан класс `.nc-hidden` для скрытия (чтобы не было моргания интерфейса до выполнения логики JS).

### Шаг 3: Клиентская обработка и проверка Cookie (Vanilla JS / Browser logic)
*Архитектурное решение: Почему мы не проверяем Cookie на сервере (PHP)? При использовании PHP (`isset($_COOKIE['...'])`), кэширующие плагины (WP Rocket, LiteSpeed, Cloudflare) могут "запомнить" страницу без баннера и отдавать ее новым посетителям. В реалиях WordPress проверка куки согласия обязана быть исключительно клиентской (браузерной) — для 100% совместимости с любым кэшем.*

1.  Браузер загружает HTML и JS. Срабатывает событие `DOMContentLoaded`.
2.  JS скрипт плагина ищет куки согласия (например, `nc_accepted=1`).
3.  Если Cookie **нет**:
    *   С плашки `div#nc-cookie-banner` удаляется класс `.nc-hidden`. 
    *   Активируется CSS transition: плавное появление из нижнего левого угла.
4.  Если пользователь нажимает кнопку "ОК":
    *   JS устанавливает cookie `nc_accepted=1` сроком на `30 дней` (`max-age=2592000; path=/`).
    *   Плашке возвращается класс `.nc-hidden` (запускается плавная CSS-анимация скрытия).
    *   Срабатывает лисенер завершения анимации, который окончательно убирает блок с экрана (через изменение видимости или перенос из DOM), чтобы он не закрывал кликовые области.

## 4. Ограничения и Безопасность
*   Экранирование вывода: все данные, берущиеся из БД/опций (например, URL политики) проходят `esc_url()`. Тексты локализации обязательно обрабатываются через `esc_html__()`.
*   Отсутствие БД: плагин не создает собственных таблиц и не сохраняет лишние значения в `wp_options`.
*   Zero Dependencies: CSS и JS должны быть микроскопическими и не тянуть за собой фреймворков.
